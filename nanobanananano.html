第1章：言語の目的と哲学

1.1 ナノバナナナノとは

ナノバナナナノは、漫画制作に特化した、完全な論理性と厳格な構文を持つプログラミング言語です。その究極の目的は、人間に代わってAIが、創造的かつ再現可能な漫画を生成するための論理的基盤を提供することにあります。この言語は、絵柄、キャラクター、ストーリー、そして感情といった漫画のあらゆる要素を、誰が見ても同じ解釈になるよう、厳密に定義されたコードに変換します。

1.2 哲学：論理的思考の絶対性

ナノバナナナノの核心は、「あいまいさの絶対的な排除」にあります。自然言語の持つ多義性や主観性を排し、すべての命令は予測可能で一意の結果を導き出します。もし構文に少しでも矛盾やあいまいさが存在すれば、それはただちにエラーとして認識され、実行は停止します。この厳格なルールは、AIが思考プロセスにおいて迷うことを許さず、与えられたプロンプトに忠実に、完璧な漫画を生成することを保証します。

第2章：基本構文と実行順序

2.1 基本的な構文

ナノバナナナノは、Pythonに類似した厳格な構文に基づいています。すべてのコードは、特定のルールに従って記述されなければなりません。

  • キーワード: global, page, frame, character, dialogue, background, effect, animate, templateなどの予約語は、命令の開始を意味します。
  • 識別子: [名前]の形式で、ユーザーが定義する要素（例：[ユキ]）を示します。これは常に一意でなければなりません。
  • 辞書型引数: すべての命令の詳細は、{ キー: 値, ... }という形式で記述します。キーと値の間はコロン（:）、各ペアはカンマ（,）で区切ります。最後のペアの後にカンマは不要です。
  • 文字列: 文字列は必ずダブルクォーテーション（"）で囲みます。
  • コメント: #で始まる行は、すべてコメントとして扱われ、実行には影響しません。

2.2 実行順序の絶対性

ナノバナナナノのコードは、以下の厳格な順序で実行されます。この順序を逸脱した場合、コンパイラは実行時エラーを返します。

  1. グローバル設定の定義: コードの最上部に、作品全体で共有される変数（globalキーワードと$プレフィックスを使用）を定義します。
  2. キャラクター定義の記述: グローバル設定の直後に、作品に登場するすべてのキャラクターを@キャラクター定義で一括して記述します。
  3. ページの作成: ストーリーの本体であるpageを記述します。pageは必ず定義済みのグローバル設定やキャラクターを参照します。

この順序は、AIが思考プロセスにおいて階層的なデータ構造を構築し、物語全体の一貫性を保つために不可欠です。

2.3 エラー処理の原則

ナノバナナナノは、あいまいさを許容しません。以下の状況では、必ずエラーが発生し、修正を求めます。

  • 構文エラー: 構文ルールに従っていない場合。
  • 型エラー: 変数や関数の引数に、宣言された型と異なる型の値を渡した場合。
  • 名前の一意性エラー: 同じ名前の変数、キャラクター、テンプレートを複数回定義した場合。
  • 未定義エラー: 存在しない変数、キャラクター、テンプレート、または型を参照した場合。

第3章：型システムとグローバル変数

3.1 型システムの絶対性

ナノバナナナノは、強力な型システムを採用しています。これにより、AIがどの情報が何のために使われるかを明確に区別できます。主な型は以下の通りです。

  • string: テキストデータ。例: "ユキ", "夕焼け"。
  • integer: 整数。例: 1, 10。
  • coordinates: 8x16グリッド上の単一の座標。例: 50。
  • coordinate_range: 座標の範囲。例: "1-32"。
  • mood: 定義済みの雰囲気。許容される値: "シリアス", "コメディ"など。
  • style: 定義済みの絵柄。許容される値: "鉛筆画", "アニメ風"など。
  • lighting: 定義済みの光源。許容される値: "逆光", "順光"など。
  • shot: 定義済みの構図。許容される値: "long_shot", "close_up"など。
  • expression: 定義済みの感情。許容される値: "exaltation" (歓喜)、"melancholy" (憂鬱)など。

3.2 グローバル変数の定義と利用

グローバル変数は、作品全体で共通の設定を格納します。これらは、globalキーワードと$プレフィックスを用いて、コードの最上部に一度だけ定義します。

# グローバル変数の宣言と型指定
global $art_style: style = "アニメ風"
global $background_type: string = "公園"
global $scene_mood: mood = "シリアス"

# 作品の途中でのグローバル変数の再定義は許可されない
global $art_style: style = "水彩画"  # <- エラー: 再定義は許可されていません

3.3 型不一致のエラー処理

型システムは、意図しないデータの使用を厳しく取り締まります。例えば、mood型にinteger型の値を代入しようとすると、コンパイラは即座に型エラーを返します。

# エラー: `mood`型に`integer`型の値は代入できません
global $scene_mood: mood = 5  # <-- エラー

# 修正後: 正しい型の値を代入
global $scene_mood: mood = "コメディ" # OK

第4章：キャラクター定義とインスタンス

ナノバナナナノにおけるキャラクターは、作品全体で一貫性を保つための厳格なクラスです。すべてのキャラクターは、@キャラクター定義構文を用いて、コードのキャラクター定義の記述セクションで一括して宣言されます。

4.1 @キャラクター定義の構文

キャラクターの定義には、以下の厳格なルールが適用されます。

# 正しい定義
@キャラクター定義 [ユキ]{
    名前: "ユキ",
    性格: "元気",
    容姿: "ショートカットの黒髪、制服"
}

# エラー: 識別子が重複しているため、定義を修正してください。
# @キャラクター定義 [ユキ]{
#     名前: "ユキ",
#     性格: "論理的"
# }

4.2 キャラクターインスタンスの作成

@キャラクター定義で宣言されたキャラクターを、実際にコマ内に配置する際には、character構文を使用します。必須引数は、一意の識別子、位置、構図、感情の4つです。

# コマ内にユキのインスタンスを配置
frame {
    座標: "1-32"
    character [ユキ] {
        位置: 10,
        構図: "close_up",
        感情: "surprise_positive"
    }
}

# エラー: 定義されていないキャラクターをインスタンス化しようとしています。
# character [ヒロシ]{ # <- エラー: `@キャラクター定義 [ヒロシ]`が存在しません
#    位置: 20,
#    構図: "medium_shot",
#    感情: "笑顔"
# }

第5章：ページの構造

ナノバナナナノの物語は、pageと呼ばれる単位で構成されます。各pageは、独立したウィンドウとして機能し、frameやdialogueといった下位の要素を内包する親コンテナです。

5.1 pageの構文と階層構造

page [ページ番号] {キー: 値, ...}の構文を使用します。ページ番号はinteger型で、1から始まる昇順に記述しなければなりません。

# 正しいページ構造
page [1]{
    # このページに適用される設定
    フィルター: "sepia" # string型
    # このページ内のコマ
    frame {
        座標: "1-64"
        # コマ内の要素
    }
    frame {
        座標: "65-128"
        # コマ内の要素
    }
}

# エラー: `page`の外に`frame`は配置できません
# frame { # <- エラー: `frame`は`page`の中に記述する必要があります
#    座標: "1-32"
# }

5.2 pageの属性

pageは、ページ全体に適用される設定を持つことができます。例：フィルター ("monochrome", "sepia"など)、照明 (lighting型)、気分 (mood型)、絵柄 (style型)。

第6章：コマの配置と座標システム

ナノバナナナノのすべてのページは、物理的なサイズに関わらず、論理的に8x16のグリッド（総マス数128）として扱われます。このグリッドは、左上を1、右下を128とする絶対座標でナンバリングされています。

6.1 8x16グリッドシステムの絶対性

このグリッドは、漫画ページの「設計図」として機能し、主観的な指示を排除します。frameは、coordinate_range型で定義された、開始座標と終了座標のペアを引数として受け取り、コマの長方形の領域を厳密に定義します。

# ページ全体を占める大きなコマ
frame {
    座標: "1-128"
    # コマ内の要素
}

# 左上隅の小さなコマ
frame {
    座標: "1-4"
    # コマ内の要素
}

第7章：背景と情景描写

背景はbackground関数で定義し、詳細な情報を提供します。必須キーはタイプと詳細の二つです。

background {
    タイプ: "森",
    詳細: "鬱蒼とした針葉樹林。地面には分厚い苔が生えており、霧が立ち込めている。木々の間からわずかに光が差し込み、神秘的な雰囲気を醸し出している。足元には落ち葉が堆積している。",
    季節: "冬",
    時間帯: "夕暮れ時",
    天気: "雪が降っている"
}

詳細には、背景の具体的な情景を非常に長く、詳細に記述します。あいまいな表現は厳禁です。

第8章：セリフと吹き出し

セリフや心の声はdialogue関数で表現します。必須引数は、キャラクター、内容、位置です。

8.1 dialogue関数の構文

dialogue {
    キャラクター: @ユキ,
    内容: "ああ、よかった！",
    位置: 40
}

8.2 吹き出しのタイプ

タイプ引数を使用することで、感情的なニュアンスを付加できます。例："叫び"、"独り言"、"小声"、"ロボット"。

# 叫び声の例
dialogue {
    キャラクター: @ユキ,
    内容: "うそ！信じられない！",
    位置: 50,
    タイプ: "叫び"
}

8.3 文字装飾のオプション

フォント、サイズ、強調といったオプションで文字を調整できます。強調は、"bold"、"underline"、"italic"などを指定可能です。

第9章：構図と感情表現

キャラクターの構図と感情は、characterインスタンス内で厳密に定義します。

9.1 構図（shot）

shot型は、キャラクターと背景の相対的なサイズやカメラアングルを決定します。例："long_shot"、"medium_shot"、"close_up"、"dutch_angle"など。

9.2 感情表現（expression）

expression型で、キャラクターの表情を具体的に指定し、あいまいな表現を排除します。例："exaltation" (歓喜)、"melancholy" (憂鬱)、"disgust" (嫌悪感)など。

character [ユキ] {
    位置: 10,
    構図: "dutch_angle",
    感情: "fear"
}

9.3 キャラクターの向き（direction）

向き引数（string型）で、キャラクターが向いている方向を指定できます。例："front"、"back"、"left"、"right"。

第10章：付録と応用

10.1 アニメーションと特殊効果

effect関数とanimate関数を使用し、短い動きや視覚効果を表現できます。effectは"flash"や"shake"といった効果を適用し、animateはキャラクターの表情やポーズの変化を時間軸で記述します。

# ユキの表情が0.5秒かけて笑顔から困惑に変化
animate { 対象: @ユキ, 変化: "笑顔->困惑", 期間: "0.5s" }

10.2 テンプレート機能と再利用性

template関数で再利用したいコードブロックを定義し、call_templateで呼び出します。これにより、コードの重複を避け、効率を向上させることができます。

# テンプレートの呼び出し
page [1]{
    call_template [二人の会話] {
        キャラA: ユキ,
        キャラB: アキラ,
        背景タイプ: "カフェ",
        背景詳細: "窓から光が差し込む賑やかなカフェ",
        感情A: 笑顔,
        セリフA: "元気？",
        感情B: 無関心,
        セリフB: "まあね。"
    }
}
